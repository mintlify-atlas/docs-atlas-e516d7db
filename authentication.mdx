TaskForge API uses JSON Web Tokens (JWT) to authenticate requests. You get two tokens when you log in — a short-lived access token for API calls, and a long-lived refresh token to get new access tokens without logging in again.

<Warning>
  Store tokens securely. Never expose them in client-side code, browser local storage, or logs. In production, use secure, HTTP-only cookies or a secure secrets manager. Anyone with a valid token can act as that user until the token expires or is revoked.
</Warning>

## How tokens work

| Token | Lifetime | Purpose |
|---|---|---|
| Access token | 1 hour (3600 seconds) | Authenticates individual API requests |
| Refresh token | 30 days (2,592,000 seconds) | Issues new access tokens without re-login |

Refresh tokens are stored in the database, which means they can be individually revoked — for example, on logout or password change.

## Auth flow

1. You call `POST /api/auth/register` to create an account.
2. You call `POST /api/auth/login` with your email and password.
3. The API returns an `access_token` and a `refresh_token`.
4. You include the `access_token` in the `Authorization` header on every protected request.
5. When the access token expires, you call `POST /api/auth/refresh` with your `refresh_token` to get a new access token.
6. On logout, you call `POST /api/auth/logout` to revoke the refresh token.

## Register

Create a new user account. All new users are assigned the `user` role by default.

**Rate limit:** 5 requests per hour per IP.

```http
POST /api/auth/register
Content-Type: application/json
```

**Request body:**

| Field | Type | Required | Description |
|---|---|---|---|
| `username` | string | Yes | Unique username |
| `email` | string | Yes | Unique email address |
| `password` | string | Yes | Must meet password complexity requirements |
| `first_name` | string | No | User's first name |
| `last_name` | string | No | User's last name |

```json
{
  "username": "johndoe",
  "email": "john@example.com",
  "password": "SecurePass123!",
  "first_name": "John",
  "last_name": "Doe"
}
```

**Success response** `201 Created`:

```json
{
  "success": true,
  "message": "Usuario registrado con exito",
  "data": {
    "id": 1,
    "username": "johndoe",
    "email": "john@example.com",
    "first_name": "John",
    "last_name": "Doe",
    "full_name": "John Doe",
    "is_active": true,
    "created_at": "2026-03-17T12:00:00",
    "updated_at": "2026-03-17T12:00:00",
    "role": { "id": 2, "name": "user", "description": null }
  }
}
```

**Error responses:**

| Status | Cause |
|---|---|
| `400` | Missing required fields or validation failure |
| `409` | Username or email already in use |

## Log in

Authenticate with your email and password to receive tokens.

**Rate limit:** 10 requests per hour per IP.

```http
POST /api/auth/login
Content-Type: application/json
```

**Request body:**

```json
{
  "email": "john@example.com",
  "password": "SecurePass123!"
}
```

**Success response** `200 OK`:

```json
{
  "success": true,
  "message": "Inicio de sesion exitoso",
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "refresh_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "user": {
      "id": 1,
      "username": "johndoe",
      "email": "john@example.com",
      "first_name": "John",
      "last_name": "Doe",
      "full_name": "John Doe",
      "is_active": true,
      "role": { "id": 2, "name": "user", "description": null }
    }
  }
}
```

**Error responses:**

| Status | Cause |
|---|---|
| `401` | Invalid email or password |
| `403` | Account is inactive |

## Make authenticated requests

Pass the `access_token` in the `Authorization` header using the `Bearer` scheme:

```http
GET /api/tasks
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

```bash
curl -X GET http://localhost:5000/api/tasks \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN"
```

<Info>
  If you omit the `Authorization` header or provide an expired token, the API returns `401 Unauthorized`.
</Info>

## Refresh your access token

When your access token expires, use your refresh token to get a new one without logging in again. Pass the refresh token in the `Authorization` header.

```http
POST /api/auth/refresh
Authorization: Bearer YOUR_REFRESH_TOKEN
```

**Success response** `200 OK`:

```json
{
  "success": true,
  "message": "Token refrescado con exito",
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
  }
}
```

**Error responses:**

| Status | Cause |
|---|---|
| `401` | Refresh token is invalid, expired, or revoked |

## Log out

Revoking your refresh token ensures it can no longer be used to issue new access tokens. Pass your current access token in the header and provide the refresh token in the request body.

```http
POST /api/auth/logout
Authorization: Bearer YOUR_ACCESS_TOKEN
Content-Type: application/json
```

**Request body:**

```json
{
  "refresh_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
}
```

**Success response** `200 OK`:

```json
{
  "success": true,
  "message": "Sesion cerrada con exito"
}
```

## Change your password

Changing your password automatically revokes all existing refresh tokens for your account, forcing re-authentication on all devices.

**Rate limit:** 3 requests per hour per IP.

```http
POST /api/auth/change-password
Authorization: Bearer YOUR_ACCESS_TOKEN
Content-Type: application/json
```

**Request body:**

```json
{
  "old_password": "SecurePass123!",
  "new_password": "NewSecurePass456!"
}
```

**Success response** `200 OK`:

```json
{
  "success": true,
  "message": "Contrasena cambiada con exito"
}
```

**Error responses:**

| Status | Cause |
|---|---|
| `400` | Missing fields or new password fails validation |
| `401` | Current password is incorrect |

## Role-based access control

Every user has one of two roles. The role determines which endpoints and resources they can access.

| Role | Description |
|---|---|
| `user` | Default role. Can create and manage their own tasks and tags. Cannot access other users' data. |
| `admin` | Full access. Can read, update, and delete any user's tasks. Can manage all users. |

### What each role can access

**User role:**
- Create, read, update, and delete their own tasks
- Create and manage their own tags
- View and update their own profile
- Export their own tasks

**Admin role:**
- Everything the user role can do
- List and manage all users
- Access tasks belonging to any user
- Deactivate or delete user accounts

<Note>
  New accounts always receive the `user` role on registration. Assigning the `admin` role requires direct database access or an existing admin account using the user management endpoints.
</Note>

### Permission errors

If you try to access a resource you don't have permission for, the API returns:

```json
{
  "success": false,
  "error": "Forbidden",
  "message": "You do not have permission to access this resource"
}
```

Attempting to access another user's resource as a regular user returns:

```json
{
  "success": false,
  "error": "Forbidden",
  "message": "You can only access your own resources"
}
```

## Rate limits summary

| Endpoint | Limit |
|---|---|
| `POST /api/auth/register` | 5 per hour |
| `POST /api/auth/login` | 10 per hour |
| `POST /api/auth/change-password` | 3 per hour |
| All other endpoints (default) | 200 per day, 50 per hour |
